# Image recognition — design & roadmap

Goal: read a Gothic 1 Remake lockpicking screenshot and auto-fill the solver's
**mechanism count (N)** and each **mechanism's position (0–6)**. The interaction
matrix is *not* read from the image (it's behavioral, not visible in one frame).

This is a three-phase effort. **Phase 1 (data collection) is implemented** on the
`feature/image-recognition` branch. Phases 2–3 are specified here so they can be
built later without re-deriving decisions.

---

## Architecture decisions (locked)

| Topic | Decision | Why |
|-------|----------|-----|
| Inference runtime | `onnxruntime-web` (WASM/WebGPU) in the webview | One model runs in **both** the website and the Tauri app; serverless; works on the static AWS site. |
| Model shape | Whole-lock-region CNN with **N position heads** (7-way) + an "absent" class | Handles the angled/overlapping 3D plates and variable mechanism counts without per-plate cropping. |
| Capture | Game window by title (Rust `xcap`) | Consistent framing; background still varies by chest. |
| Labels | Auto-generated by driving the lock from a known reset state | No manual annotation — the big advantage over the ability-draft project. |
| Training | Colab, MobileNetV2 transfer learning → INT8 ONNX | Same proven toolchain as `ability-draft-plus`. |

Reference project studied: `ability-draft-plus` (MobileNetV2, 96×96, INT8 ONNX,
`onnxruntime-node`, automated data capture via game console commands).

## Phase 1 — Data collection (DONE)

In the desktop app, the **🧪 Data collection (dev)** panel (visible only when
running as the Tauri app) drives the lock and saves labeled screenshots.

### How it works
1. You set up one real lock in the normal UI: **N**, the **interaction matrix**,
   and set **Starting positions to the lock's reset state** (what pressing `R`
   produces). Confirm the solver actually opens this lock — if it does, the
   matrix is correct and generated labels will be correct.
2. Set an output folder, sample count, and random-walk length, then **Start**.
3. For each sample the harness: presses the reset key → applies a short
   **random** solver-tracked move sequence → waits → **captures the game window**
   → writes `<name>.png` + `<name>.json`.

Why random walks (not exhaustive): the full state space is 7^N (≈118k for N=6).
Random sampling covers each mechanism×position richly at a fraction of the cost.
Resetting every sample prevents a dropped keypress from corrupting later labels.

### Backend commands (`src-tauri/src/lib.rs`)
- `focus_window(title)` — bring the game to the foreground.
- `send_keys(keys, holdMs, delayMs)` — send key tokens immediately (scancodes).
- `save_sample(title, dir, name, labelJson)` — capture the window via `xcap`,
  save `<dir>/<name>.png`, and write `<dir>/<name>.json` (label + image `width`/`height`).

### Dataset format
Each sample is a PNG plus a sidecar JSON:

```json
{
  "n": 6,
  "positions": [3, 1, 4, 3, 0, 5],
  "reset":     [3, 3, 3, 3, 3, 3],
  "width": 1920,
  "height": 1080
}
```

- `positions[i]` ∈ `[0,6]` — the hole aligned with the keyway for mechanism `i`
  (index 3 = centered/solved).
- `reset` — the baseline positions for that lock (provenance/debugging).
- `width`/`height` — captured window size (resolution bucket).

### Collection plan for a good dataset
- **Multiple locks**: vary N (e.g. 3–7) and chest/background. The fixed
  foreground generalizes; backgrounds must vary so the model ignores them.
- **Multiple resolutions**: change the game resolution, re-run into a new folder
  (e.g. `1920x1080/`, `2560x1440/`, `1600x900/`).
- **Volume**: aim for ~1–3k samples per (lock, resolution). Random walks give
  good marginal coverage of each mechanism×position.
- **Sanity**: spot-check a few `.png`/`.json` pairs — does the image match the
  labeled positions?

## Phase 2 — Training (TODO; Colab)

Single reproducible notebook + `requirements.txt` (cleaner than the 17-step
scripts in the reference project).

### Preprocessing (MUST match inference exactly)
1. Load each PNG. Crop a **generous fixed region** around the lock (right portion
   of the frame). The exact box is a fraction of `width`/`height` so it's
   resolution-independent; tune once against the dataset.
2. Resize to a fixed input — start with **160×96** (wider than tall, matching the
   lock's shape) or 128×128; decide empirically.
3. Normalize with the MobileNetV2 convention (`Rescaling(1./127.5, offset=-1)`),
   ideally as a layer baked into the exported model so JS only feeds 0–255 RGB
   (the reference project does this — avoids normalization drift).

### Model
- Backbone: **MobileNetV2** (ImageNet, frozen initially; optionally fine-tune
  later layers).
- Head: GlobalAveragePooling → Dropout(0.2) → **MAX_MECHANISMS parallel Dense
  heads**, each `softmax` over **8 classes** = positions 0–6 plus `absent`.
  - `MAX_MECHANISMS = 7` (the game's max).
  - For a lock with N<7, heads `N..6` are labeled `absent` (class 7).
- Loss: sum of per-head `SparseCategoricalCrossentropy`. Metric: per-head and
  exact-match (all heads correct) accuracy.

### Label encoding
For a sample with `positions = [3,1,4,3,0,5]` (N=6), the target vector is
`[3,1,4,3,0,5, 7]` (head 6 = absent). Mask/encode consistently for all N.

### Augmentation (critical for robustness)
- **Translation & scale jitter** so the model tolerates an imperfect crop (this
  is what lets us skip per-resolution coordinate tables).
- **Brightness/contrast/hue** jitter — dungeons are dark and torch-lit.
- **Background augmentation**: optionally composite the cropped foreground over
  random backgrounds, or darken outside a soft mask, so the model ignores the
  chest/room. (User idea; strong lever.)
- **No horizontal flip** — it would invert positions. (The reference project
  flips because icon identity is flip-invariant; ours is not.)

### Export
1. Save an inference model (augmentation removed, BN in inference mode) as a
   TF SavedModel.
2. `tf2onnx --saved-model … --output lock_recognizer.onnx --opset 18`.
3. INT8 dynamic quantization (`onnxruntime.quantization.quantize_dynamic`).
4. Validate INT8 exact-match accuracy on a held-out test split before shipping.

### Outputs to commit
- `resources/model/lock_recognizer_int8.onnx`
- `resources/model/recognizer_meta.json` — input size, crop fractions, class
  layout (`MAX_MECHANISMS`, `absent` index), normalization convention.

## Phase 3 — Inference integration (TODO; app + web)

Add `onnxruntime-web` to `web/` (vendored or via CDN-bundled asset, since the
site is a single static file). Implement a shared module:

### Shared inference path (runs in the webview, both web and app)
1. Obtain an image:
   - **Desktop app**: a new Rust command captures the game window and returns the
     PNG bytes (or reuse the capture path and hand pixels to JS).
   - **Website**: user **pastes/drops** a screenshot (or optional
     `navigator.mediaDevices.getDisplayMedia` for a live grab).
2. Crop the lock region using the same fractions as training (`recognizer_meta`),
   resize to the model input on a `<canvas>`, read RGBA → RGB Float32 (0–255).
3. `ort.InferenceSession` (`['webgpu','wasm']` providers) → run → per-head argmax.
4. Map heads: `absent` → mechanism not present (sets N); otherwise position 0–6.
5. Fill the solver UI: set N (count of non-absent leading heads), set positions,
   re-render. User reviews, then Solves / Executes.

### UI
- A **"Read from screenshot"** button near section 1.
  - App: triggers capture + inference automatically.
  - Web: opens a paste/drop target, then inference.
- Show a confidence readout per mechanism; flag low-confidence heads for manual
  correction (mirrors the reference project's 0.9 threshold + manual review).

### Notes
- Keep the model + `recognizer_meta.json` next to `index.html` so the static
  site can fetch them.
- The desktop app already bundles `web/`, so the same files ship in both.

## Phase 4 — Auto-calibrate the matrix (future, optional)

With capture + recognizer in place, the matrix can be inferred automatically:
from reset, nudge mechanism `i` one step, recognize the new positions, and read
off how every mechanism changed (same / opposite / none). Repeat per mechanism.
This would fully automate setup and is a strong portfolio demo. The Phase 1
harness and the Phase 3 recognizer are exactly the pieces it needs.
